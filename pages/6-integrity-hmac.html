Lesson 6

Integrity checking & HMAC

Same input → same hash. So a hash is a fingerprint you can use to prove a file hasn't changed. Add a secret key and you also prove who made it.

File integrity

Download an ISO, hash it, compare to the published hash. Match = your copy is byte-identical to the official one. One flipped bit → a completely different hash (avalanche again).

$ sha256sum Fedora-Workstation-Live-x86_64-40-1.14.iso
dd1faca950d1a8c3d169adf2df4c3644ebb62f8aac04c401f2393e521395d613  Fedora-...iso

Uses beyond downloads: spotting corruption, and finding duplicate files (same hash = same file).

Try it — integrity check

Type "file contents" below. If the received copy matches the expected hash, it's intact. Flip one character to simulate corruption and watch it fail.

HMAC — integrity and authenticity

A plain hash proves integrity but not who sent it — anyone can recompute it. An HMAC (Keyed-Hash Message Authentication Code) mixes in a secret key, so only someone with the key can produce a valid tag. It proves both:

  • Integrity — the message wasn't modified.
  • Authenticity — it came from someone who holds the key.

The formula, where K = key, M = message, H = hash:

HMAC(K,M) = H( (K ⊕ opad) || H( (K ⊕ ipad) || M ) )

Roughly: pad the key to the block size, XOR it with two constants (ipad/opad), and hash twice — inner hash over the message, outer hash over the inner result.

Try it — live HMAC

hashcat & HMAC hashcat cares whether the salt/key is the "pass" or the "salt". For HMAC-SHA512 (key = $pass) the mode is 1750; HMAC-SHA512 (key = $salt) is 1720. Read the mode name carefully.
