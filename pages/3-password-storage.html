Lesson 3

Storing passwords the right way

A server should never know your password. It should store only a salted hash, so a database leak doesn't instantly hand over everyone's credentials.

How login actually works

You never need to recover a password to log someone in — you only need to check they know it. So the server stores hash(password). At login it hashes what you typed and compares. Match → you're in. The plaintext is never stored.

Authentication ≠ password manager A password manager must give you back the cleartext, so it uses encryption. Authentication only confirms knowledge, so it uses hashing. Different problems, different tools.

Three ways companies got it wrong

BreachMistakeWhy it's bad
RockYou (2009)Plaintext passwords14M+ passwords leaked as-is → the famous rockyou.txt wordlist.
Adobe (2013)Reversible encryption + plaintext hintsOne key decrypts everything; hints revealed passwords.
LinkedIn (2012)Unsalted SHA-1Fast, broken algorithm + no salt = rainbow-table heaven.

The salt problem

If two users share a password, an unsalted hash stores the same digest for both. Crack one, crack both. Worse, attackers can precompute a rainbow table (a giant hash→plaintext lookup) once and reuse it forever.

A salt is a random, per-user value added to the password before hashing. Now identical passwords get different hashes, and generic rainbow tables are useless.

See it: same password, different salts

Doing it properly 1. Pick a slow, salted algorithm: Argon2, scrypt, bcrypt, or PBKDF2. 2. Generate a unique random salt per user. 3. Hash password + salt. 4. Store the hash and the salt (salts don't need to be secret). bcrypt/scrypt/Argon2 do the salting for you.
Why not just encrypt passwords? Encryption needs a key you must store somewhere. Steal the key → decrypt every password. Hashing has no key to steal. For authentication, the answer is Nay — don't encrypt, hash.
