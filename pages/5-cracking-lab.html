Lesson 5

Cracking lab

You can't "decrypt" a hash. You crack it — hash millions of guesses (adding the salt if present) until one matches. hashcat and John the Ripper do the heavy lifting.

Legal & safe note The terminal below simulates cracking for the room's known hashes so you can learn the workflow without any real cracking. Only crack hashes you're authorised to (CTFs, your own systems, engagements).

hashcat syntax

hashcat -m <hash_type> -a <attack_mode> hashfile wordlist
PartMeaning
-m <n>Hash type number (e.g. 0 = MD5, 1000 = NTLM, 1400 = SHA256, 3200 = bcrypt).
-a <n>Attack mode. -a 0 = straight (wordlist).
hashfileFile containing the hash(es).
wordliste.g. /usr/share/wordlists/rockyou.txt.

Example: hashcat -m 3200 -a 0 hash1.txt /usr/share/wordlists/rockyou.txt

John the Ripper

john --wordlist=/usr/share/wordlists/rockyou.txt hash.txt
john --show hash.txt

John auto-detects many formats and uses CPU by default — it works in a VM out of the box.

GPU vs CPU (and VMs)

  • GPUs have thousands of cores → fast at most hashes. Run hashcat on your host for GPU speed.
  • bcrypt is deliberately GPU-unfriendly — that's a feature, it resists cracking.
  • VMs usually can't reach the host GPU; expect slower speeds inside one. John on CPU is fine in a VM.
