Lesson 7

Hashing vs encoding vs encryption

These three get confused constantly. The one-line test: hashing is one-way, encoding needs no key, encryption needs a key.

The three, side by side

HashingEncodingEncryption
Reversible?NoYesYes
Needs a key?NoNoYes
PurposeFingerprint / integrity / password storageCompatibility / safe transportConfidentiality
ExamplesMD5, SHA-256, bcryptASCII, UTF-8, Base64, Base32AES, RSA
Output sizeFixedVaries with inputVaries with input
Base64 is NOT encryption Base64 hides nothing — anyone can decode it with no key. If you see == at the end of a string, think Base64, not "encrypted".

Encoding is for compatibility, not secrecy

Encodings represent data in a form a system can handle. Character encodings (ASCII, UTF-8/16/32) map characters to bytes — UTF-8/16/32 can represent any language. Binary-to-text encodings (Base64, Base32) pack arbitrary bytes into safe printable characters for transport in email, URLs, JSON, etc.

The mental model

  • Hashing — "prove it's the same / prove you know it" — no going back.
  • Encoding — "make it fit / travel safely" — reverse freely, no key.
  • Encryption — "keep it secret" — reverse only with the key.
